# 文史哲AI多模块生成系统（面向AI编程工具的PRD文档）

## 1. 项目目标（Objective）
构建一个统一的文史哲AI生成平台，将以下10个独立功能模块整合为一个可切换的AI应用系统。系统通过统一入口接收用户输入，根据模块类型调用不同的提示词逻辑生成结果。

目标：
- 支持10个文史哲AI生成子系统
- 支持统一用户界面（单页应用）+ 模块选择
- 每个模块输出结构化结果
- 支持后续扩展新模块

---

## 2. 系统整体架构（Architecture）

### 2.1 系统结构
- 前端：单页应用（SPA）
- 后端：API服务（Node.js / Python均可）
- 大模型层：统一提示词路由器 + 模块提示词库

### 2.2 核心流程
用户输入 → 选择模块 → 输入参数 → 提示词路由 → 大模型生成 → 结构化输出 → 前端渲染

---

## 3. 模块统一规范（Module规范标准）

每个模块必须符合以下接口规范：

### 模块接口定义
```json
{
  "模块ID": "string",
  "模块名称": "string",
  "输入结构": {},
  "输出结构": {},
  "提示词模板": "string"
}
```

### 输出要求
- 必须结构化输出（优先JSON，其次Markdown分段）
- 禁止无结构长文本输出
- 每个模块必须具有固定输出字段

---

## 4. 模块定义（10个核心模块）

说明：
每个模块除了输入输出结构之外，还必须绑定对应的“核心提示词模板（Prompt Template）”。
AI编程工具必须将提示词模板作为独立配置项管理，不允许硬编码在业务逻辑中。

所有提示词模板必须支持：
- 参数注入
- 模块化替换
- 后续扩展
- 动态Prompt优化

---

### 4.1 典故时间穿越解释器

模块ID：idiom_time_travel

输入：
- 典故：string
- 历史时期列表：array[string]

输出：
- 各时期分析数组
  - 时期
  - 字面含义
  - 历史语境
  - 解释方式
  - 与现代差异

核心提示词模板：
```text
你是一位历史语义学家与文化解释者。

任务：
对一个典故进行“跨历史时期语境重构解释”。

输入：
典故：{idiom}
历史时期：{eras}

要求：
1. 对每个历史时期分别输出：
   - 字面理解
   - 社会语境
   - 价值观解读
   - 与现代差异
2. 输出必须结构化
3. 禁止泛泛而谈
4. 使用学术性表达
```

---

### 4.2 哲学家群聊模拟器

模块ID：philosopher_chat

输入：
- 哲学家列表：array[string]
- 问题：string

输出：
- 对话记录数组
  - 发言者
  - 内容
  - 类型（陈述/反驳/提问）
- 总结

核心提示词模板：
```text
你是一位哲学史研究者，同时扮演多位哲学家进行群聊模拟。

参与角色：
{philosophers}

讨论问题：
{question}

要求：
1. 每位哲学家必须保持思想一致性
2. 必须包含观点冲突
3. 至少出现一次反驳
4. 输出格式必须类似聊天记录
5. 最后输出“未达成共识总结”
```

---

### 4.3 历史反事实模拟器

模块ID：counterfactual_history

输入：
- 历史事件：string
- 变量变化：string

输出：
- 原始历史概述
- 分支结果数组
  - 分支名称（保守/激进/黑天鹅）
  - 时间线数组
    - 年份
    - 事件

核心提示词模板：
```text
你是一位历史推演建模专家。

任务：
进行反事实历史推演。

输入：
历史事件：{event}
变量变化：{changed_variable}

要求：
1. 先描述真实历史路径
2. 再描述变量改变后的直接影响
3. 必须生成三条不同发展分支：
   - 保守路径
   - 激进路径
   - 黑天鹅路径
4. 每条路径必须包含时间线节点
```

## 5. 提示词路由器（核心调度逻辑）

系统必须根据模块ID调用对应提示词：

```pseudo
函数 route(模块ID, 输入):
    提示词 = 加载提示词(模块ID)
    完整提示词 = 注入参数(提示词, 输入)
    返回 大模型调用(完整提示词)
```

---

## 6. 通用提示词规范（所有模块必须遵守）

### 强制规则：
- 必须结构化输出
- 禁止长篇散文
- 必须字段化输出
- 必须符合输出结构定义
- 不得生成未要求内容

### 风格规范：
- 文史哲类：学术性 + 清晰 + 可解释
- 对话类：剧本格式
- 分析类：结构化报告

---

## 7. 用户界面结构建议

### 页面结构：
- 左侧：模块列表（10个模块）
- 中间：输入区域
- 右侧：结果展示区（支持JSON/Markdown切换）

---

## 8. 扩展机制（重要）

新增模块必须包含：
1. 模块ID
2. 输入结构定义
3. 输出结构定义
4. 提示词模板

系统无需修改核心代码即可扩展新模块

用户选择：哲学家群聊模拟器
